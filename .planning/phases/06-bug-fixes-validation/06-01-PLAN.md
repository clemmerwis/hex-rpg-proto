<objective>
Fix the MovementSystem setTimeout(0) race condition so movement completion callbacks fire in deterministic order.

Purpose: When multiple characters complete movement in the same frame, setTimeout(0) does not guarantee execution order. Replace with a deferred callback queue that drains synchronously after the movement update loop, ensuring callbacks fire in character array order (PC first, then NPCs in iteration order).
Output: MovementSystem with deterministic callback ordering; combat execution behaves identically but without timing-dependent callback order.
</objective>

**Race condition details (from CONCERNS.md):**
- Symptoms: Multiple movements completing same frame may fire callbacks out of order
- File: js/MovementSystem.js lines 110-117 (setTimeout 0 for async callback)
- Root cause: setTimeout(0) doesn't guarantee execution order
- Workaround: Self-heals because each callback is independent
- CombatExecutor.executeNextMove (line 171) registers movement callbacks that chain to next move

**Key insight:** The setTimeout(0) was added to avoid blocking the movement update loop. The fix must preserve this property — callbacks should not fire mid-iteration. Collecting them and draining after the loop achieves both goals: no mid-loop firing AND deterministic ordering.

  3. In `updateMovement()` (line 24), after the `allCharacters.forEach(...)` loop completes, add a drain step:
     ```javascript
     // Drain deferred callbacks (fire after all movement updates, not mid-loop)
     if (this._pendingCallbacks.length > 0) {
         // Copy and clear before draining — callbacks may trigger new movements
         // whose completions should queue for the NEXT updateMovement cycle
         const callbacks = this._pendingCallbacks;
         this._pendingCallbacks = [];
         for (const cb of callbacks) {
             try {
                 cb(true);
             } catch (error) {
                 console.error("MovementSystem: Callback error", error);
             }
         }
     }
     ```
     This ensures: (a) callbacks fire in character iteration order, (b) no callback fires during movement iteration, (c) callbacks from newly-started movements don't fire in the same drain cycle.

  4. Also add drain to `clearAllCallbacks()` — clear `_pendingCallbacks` as well so pending callbacks don't fire after combat exit.
