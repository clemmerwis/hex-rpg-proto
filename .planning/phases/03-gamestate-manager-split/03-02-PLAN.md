<objective>
Extract engagement tracking from GameStateManager into a dedicated EngagementManager module and consolidate the split engagement concern.

Purpose: Engagement tracking is currently split across GameStateManager (updateEngagement, clearNonAdjacentEngagements, tryEstablishEngagement, canEngageBack) and CombatSystem (calls canEngageBack for flanking). This is flagged in CONCERNS.md as fragile — "two code paths manage engagements independently." Extracting into a single EngagementManager creates one canonical source for all engagement logic.
Output: New js/EngagementManager.js module; GSM, CombatExecutor, and CombatSystem all reference it.
</objective>

Create js/EngagementManager.js as a new ES6 module class. Extract and consolidate engagement methods:

**Constructor signature:**
```js
constructor(hexGrid, getCharacterAtHex, logger)
```

**Methods to extract from GameStateManager (move verbatim, then adapt):**
- updateEngagement(character) — called after movement, orchestrates engagement checks
- clearNonAdjacentEngagements(character) — removes engagements for non-adjacent characters
- tryEstablishEngagement(charA, charB) — establishes mutual engagement up to engagedMax
- canEngageBack(defender, attacker) — checks if defender can engage attacker (used by CombatSystem for flanking)

**Also add cleanup method (extracted from GSM's exitCombat):**
- clearAllEngagements(pc, npcs) — clears engagedBy Sets for all characters (called on combat exit)

All methods are pure engagement logic — they read character.engagedBy, character.engagedMax, check adjacency via hexGrid.hexDistance/getNeighbors, and log via logger. No other dependencies needed.

Import nothing from const.js (engagement methods don't use game constants).
