<task type="auto">
  <name>Task 1: Create CombatExecutor.js with extracted execution methods</name>
  <files>js/CombatExecutor.js</files>
  <action>
Create js/CombatExecutor.js as a new ES6 module class. Extract these methods from GameStateManager.js:

**Methods to extract (move verbatim, then adapt):**
- sortBySpeed(characters, phase, actionsMap) — speed-tier sorting for combat ordering
- enterCombatExecution() — builds execution queue, kicks off move phase
- executeMovePhase() — filters movers, sorts by speed, starts move loop
- executeNextMove() — move execution loop with collision detection, callbacks
- executeActionPhase() — filters attackers, sorts by speed, starts attack loop
- executeNextAttack() — attack execution loop with animation timing, damage resolution
- autoFaceAdjacentEnemy(character) — post-move facing toward adjacent enemies
- handleCharacterDefeat(character) — marks defeated, removes from combat roster

**Constructor signature:**
```js
constructor(hexGrid, getCharacterAtHex, movementSystem, combatSystem, logger)
```

**Dependencies CombatExecutor needs (injected via constructor):**
- hexGrid — for hexToPixel, getNeighbors, hexDistance
- getCharacterAtHex — bound function from Game
- movementSystem — for onMovementComplete, updateFacing
- combatSystem — for executeAttack
- logger — for combat/debug logging

**State that CombatExecutor manages (owned by executor, not GSM):**
- executionQueue, currentPhase ('move'/'action')
- moveQueue, actionQueue
- currentMoveIndex, currentActionIndex

**State that CombatExecutor needs READ access to (passed in via method params or setters):**
- characterActions Map — passed to enterCombatExecution()
- combatCharacters array — passed to enterCombatExecution()
- game.pc reference — for clearing playerSelectedHex on player move

**Callback pattern for phase completion:**
- CombatExecutor.onExecutionComplete callback — GSM sets this; called when all attacks done
- CombatExecutor.onCharacterDefeated callback — GSM sets this to handle roster removal
- CombatExecutor.onClearRecentlyHit callback — GSM sets this to clear hit tracking between actions
- CombatExecutor.onClearPlayerSelection callback — GSM sets this to clear playerSelectedHex

**Method adaptations:**
- enterCombatExecution(combatCharacters, characterActions) — takes params instead of reading this.combatCharacters/this.characterActions from GSM
- executeNextMove: replace `this.game.pc` checks with callback (onClearPlayerSelection)
- executeNextMove: replace `this.clearRecentlyHitCharacters()` with callback (onClearRecentlyHit)
- executeNextAttack: replace `this.clearRecentlyHitCharacters()` with callback (onClearRecentlyHit)
- executeNextAttack: keep `this.combatSystem.executeAttack()` call (combatSystem injected)
- executeNextAttack: on all attacks done, call `this.onExecutionComplete()` instead of `this.turnNumber++; this.setState()`
- handleCharacterDefeat: call `this.onCharacterDefeated(character)` callback so GSM can splice combatCharacters

**Engagement calls stay as callbacks:**
- executeNextMove calls updateEngagement — replace with `this.onUpdateEngagement(character)` callback so GSM (which still owns engagement methods for now) handles it

**Import from const.js:**
- GAME_CONSTANTS (for COMBAT_PHASE_TRANSITION, COMBAT_ATTACK_WINDUP, COMBAT_ATTACK_RECOVERY)
- calculateMoveSpeed, calculateActionSpeed, getSpeedTier, calculateInitiative, getFacingFromDelta

Import makeEnemies from utils.js (used in executeNextAttack for hostility trigger).
